# 招聘管理系统移动端开发方案

> 本文档面向技术决策者与开发团队，分析将现有 ATS 招聘管理系统扩展至移动端的可行路径、技术选型、实施计划与风险预估。
>
> **当前状态**：待评审，尚未进入开发阶段。

---

## 1. 执行摘要（TL;DR）

| 维度 | 结论 |
|------|------|
| **推荐主方案** | **独立移动 H5（Vue 3 + Vant 4）**，新建 `mobile/` 子项目，复用现有后端 API，可无缝对接飞书 H5 微应用 |
| **备选方案** | UniApp（若未来必须同时输出微信小程序或 App） |
| **不推荐方案** | 直接改造现有 PC 端为响应式（Element Plus 移动端体验极差） |
| **预估周期** | **7～9 周**（1 名前端全职 + 0.5 名后端支持，含飞书接入） |
| **核心风险** | 后端接口未针对移动端场景做裁剪、富文本/表格在移动端的交互体验落差、飞书开放平台权限审批 |

---

## 2. 现状分析

### 2.1 现有技术栈
- **前端**：Vue 3（Composition API / `<script setup>`）+ Vite 5 + Element Plus 2.5+
- **后端**：Express 4 + Prisma 5 + PostgreSQL + JWT
- **构建**：Monorepo（pnpm workspaces），现有 `client/` + `server/` + `e2e/`
- **核心页面模块**：Dashboard、职位管理、候选人管理、面试反馈、Offer、数据统计、成员管理、系统设置

### 2.2 移动端需求的本质差异
PC 端强调**信息密度**与**复杂表格操作**；移动端强调**碎片化查看**与**快速审批/录入**。直接对现有 Element Plus 页面做响应式适配会导致：
- 表格列过多，小屏横向滚动灾难；
- 弹窗/抽屉在手机上全屏化后交互混乱；
- 表单字段过多，填写效率极低；
- 富文本编辑器（vue-quill）在移动端软键盘场景下 Bug 频出。

**结论**：移动端需要独立的页面交互设计，而非简单缩放现有 PC 页面。

## 3. 技术方案对比

### 方案 A：响应式 Web（改造现有 client）
- **思路**：在现有 `client/` 中增加断点逻辑，对 Element Plus 表格、表单做移动端适配，并打包为 PWA。
- **优点**：
  - 无新增项目，代码集中；
  - 后端 API 零改动。
- **缺点**：
  - **Element Plus 并非移动优先组件库**，移动端体验非常勉强；
  - 表格、复杂表单、富文本的改造工作量巨大，且效果上限低；
  - PC 与移动端样式耦合，后续互相牵制。
- **适用场景**：仅做简单信息展示，无复杂交互。
- ** verdict**：❌ 不推荐。

### 方案 B：独立移动 H5（Vue 3 + Vant 4）⭐ 推荐
- **思路**：在 Monorepo 中新建 `mobile/` 包，采用 Vue 3 + Vite + Vant 4（有赞移动端组件库），复用现有 `api/`、`stores/`、`utils/` 逻辑，独立设计移动 UI。
- **优点**：
  - 团队 Vue 3 技术栈 **100% 复用**，学习成本最低；
  - Vant 4 专为移动端设计，组件（List、PullRefresh、ActionSheet、Search、Uploader 等）开箱即用；
  - 与现有 PC 端 **完全解耦**，并行开发互不干扰；
  - 可部署为独立 H5，也可嵌入**飞书/企业微信/钉钉**；
    - 特别说明：飞书 H5 微应用（网页应用）是国内 ToB 最成熟的落地形态，Vant 4 在飞书内置浏览器兼容性良好；
  - 未来若需要 App，可通过 **Capacitor / Cordova** 快速打包。
- **缺点**：
  - 需要维护两套前端代码（PC + Mobile）；
  - 无法直接输出微信小程序（需另起项目或改用 UniApp）。
- **适用场景**：以手机浏览器/企业微信内嵌为主，追求开发效率与用户体验平衡。
- **verdict**：✅ **首推**。

### 方案 C：跨平台小程序/App（UniApp / UniApp-X）
- **思路**：基于 UniApp（Vue 语法）开发一套代码，输出 H5 + 微信小程序 + App（iOS/Android）。
- **优点**：
  - 一套代码覆盖多端，适合中国市场（微信小程序生态）；
  - 若公司有上线微信小程序的硬性需求，这是最优解。
- **缺点**：
  - 工程体系与现有 Vite + pnpm monorepo **差异较大**，需要额外配置；
  - 部分 npm 包在 UniApp 中兼容性存疑（如 pdf-parse、xlsx、echarts 需换用 UniApp 生态版本）；
  - 调试体验（尤其是 App 端）不如纯 Web；
  - 需要学习 UniApp 的生命周期、路由、条件编译等概念。
- **适用场景**：必须同时覆盖微信小程序或原生 App，且可接受一定生态限制。
- **verdict**：⚠️ **作为备选**，在确认有小程序刚需时启动。

### 方案 D：原生跨平台（Flutter / React Native）
- **思路**：完全抛弃 Vue 生态，用 Flutter 或 RN 重写移动端。
- **优点**：原生级体验、性能最好。
- **缺点**：
  - 与现有团队技术栈割裂，学习成本极高；
  - 开发周期长（预估 3～4 个月）；
  - 需单独维护移动端接口协议。
- **verdict**：❌ 对于当前业务阶段严重 Over-engineering，不推荐。

## 4. 推荐架构设计（方案 B 详细版）

### 4.1 仓库结构
```
recruiting-system/
├── client/               # 现有 PC 端（保持不变）
├── server/               # 现有后端（保持不变）
├── mobile/               # 新建移动端项目
│   ├── src/
│   │   ├── api/         # 复用/裁剪自 client/src/api
│   │   ├── assets/
│   │   ├── components/  # 移动端公共组件
│   │   ├── composables/ # 复用 client 中通用逻辑
│   │   ├── router/      # 移动端路由（页面更精简）
│   │   ├── stores/      # Pinia stores（与 client 共享定义）
│   │   ├── utils/       # 工具函数（尽量复用）
│   │   ├── views/       # 移动端页面
│   │   └── App.vue
│   ├── index.html
│   ├── package.json
│   ├── tsconfig.json
│   └── vite.config.ts
├── e2e/                  # 可增加 mobile-e2e 子目录
└── ...
```

### 4.2 后端 API 策略
移动端原则上**复用现有 REST API**，但需识别以下差异：

| 差异点 | 处理方式 |
|--------|----------|
| **列表分页** | 现有 API 已分页，可直接复用；但移动端建议默认 `pageSize=20` |
| **字段裁剪** | 列表接口返回字段可能过多（如 `description`、`requirements` 富文本），建议在现有接口增加 `?fields=` 查询参数，或在 BFF 层做字段过滤 |
| **图片/文件上传** | 复用现有 `/api/upload`，但移动端需调用相机/相册（使用 `<input type="file" accept="image/*">` 或 Capacitor Camera API） |
| **实时通知** | 现有系统无 WebSocket，移动端若需推送，后期可引入 SSE 或集成个推/极光推送（超出本期范围） |
| **认证** | 复用 JWT，登录后 token 存 `localStorage`，Axios interceptor 统一附加 header |

**是否新建 BFF（Backend-for-Frontend）？**
- **本期不建议**：增加一个 BFF 服务会显著提升部署复杂度。移动端直接调现有 `server` API，配合少量接口参数优化即可。
- **未来演进**：若小程序或第三方集成增多，再考虑用 NestJS/Next.js API Routes 做 BFF。

### 4.3 飞书适配设计（方案 B 的扩展）

如果目标用户主要通过**飞书**使用移动端，需在现有方案基础上增加以下适配层：

#### 4.3.1 飞书接入形态
| 形态 | 说明 | 本期是否采用 |
|------|------|-------------|
| **飞书 H5 微应用** | 在飞书开放平台创建"企业自建应用"，配置移动端首页地址指向 `mobile/` 产物。员工点击工作台图标即可在飞书内置浏览器中打开。 | ✅ **采用** |
| **飞书小程序** | 需使用飞书小程序语法（类似微信小程序 `.ttml`）重写，与 Vue 3 技术栈不兼容。 | ❌ **不采用** |
| **飞书机器人** | 后端调用飞书 OpenAPI 推送消息（如面试提醒、Offer 审批通知），与前端方案无关。 | ⚠️ **二期扩展** |

#### 4.3.2 前端需要集成的飞书能力
在 `mobile/` 中引入 `@larksuite/jsapi-sdk`：

| 能力 | 用途 | 复杂度 |
|------|------|--------|
| **飞书免登** | 用户打开应用后无需输入密码，通过 `auth_code` 换取身份，后端自动签发 JWT | 中 |
| **获取用户信息** | 获取员工姓名、头像、部门、employee_id | 低 |
| **调用原生能力** | 扫一扫、选择图片/相机、定位、文件预览 | 低 |
| **导航栏控制** | 隐藏/显示标题栏、设置右上角菜单按钮 | 低 |
| **分享** | 调用飞书原生分享，替代浏览器默认分享 | 低 |

#### 4.3.3 后端需要配合的改动
这是接入飞书时**唯一涉及现有 `server/` 的改动**，但影响面极小：

1. **新增飞书免登接口**
   - `POST /api/auth/feishu/login`
   - 流程：前端获取 `auth_code` → 传给后端 → 后端调用飞书开放平台 API 换取 `employee_id` → 匹配本地 `User` 表 → 签发 JWT
   - 若用户未绑定：可选择自动注册（内部系统推荐）或提示联系管理员

2. **数据库微调**
   ```prisma
   model User {
     // ... 现有字段
     feishuEmployeeId String? @unique
   }
   ```
   > 仅增加一个可选字段，对现有 PC 端及数据**零影响**。

3. **飞书消息推送（可选二期）**
   - 后端通过飞书 OpenAPI 发送单聊/群消息，实现面试提醒、审批待办通知等。

#### 4.3.4 飞书环境下的 UI 适配注意点
- **底部安全区**：飞书安卓版底部有系统导航条，`mobile/` 的 TabBar 必须预留 `env(safe-area-inset-bottom)`；
- **登录页隐藏**：在飞书环境下检测到已授权时，直接走免登，不展示账号密码输入框；
- **外链跳转**：简历预览/文件下载建议通过飞书 JSAPI 调用系统浏览器打开，避免内置浏览器限制；
- **分享卡片**：职位详情页可配置飞书分享，生成带封面图的卡片消息。

### 4.4 核心页面映射（PC → Mobile）
移动端页面应围绕**"看板 + 快捷操作"**重新设计，而非 1:1 复制。

| PC 模块 | 移动端定位 | 关键页面 |
|---------|-----------|----------|
| Dashboard | 今日待办看板 | 首页（待处理候选人、待面试、待审批 Offer） |
| 职位管理 | 浏览 + 转发 | 职位列表、职位详情（简化版，支持微信分享） |
| 候选人管理 | 搜索 + 状态流转 | 候选人列表、候选人详情、阶段推进、上传简历 |
| 面试反馈 | 快速填写 | 我的面试日程、填写面试反馈 |
| Offer | 审批 + 查看 | Offer 列表、Offer 审批 |
| 数据统计 | 关键指标卡片 | 数据看板（仅展示核心 KPI 图表） |
| 成员/设置 | 低频功能后置 | 个人中心、设置页 |

### 4.5 技术栈细节
| 层级 | 选型 | 理由 |
|------|------|------|
| 框架 | Vue 3.4 + TypeScript | 与 PC 端一致 |
| 构建 | Vite 5 | 与 PC 端一致，启动快 |
| 组件库 | Vant 4 | 成熟、Vue 3 官方支持、文档完善 |
| 状态管理 | Pinia 2 | 与 PC 端一致，可复用 store 类型定义 |
| 路由 | Vue Router 4 | 与 PC 端一致 |
| 图表 | VChart / ECharts 精简版 | 移动端图表需轻量 |
| CSS | vw 适配 + Vant 自带 rem 适配 | 推荐 `postcss-px-to-viewport` |
| 请求 | Axios（复用 client 封装） | 统一拦截器、错误处理 |
| 飞书 JSAPI | `@larksuite/jsapi-sdk` | 官方 SDK，支持免登、扫码、分享、导航栏控制 |
| 构建产物 | 静态 H5 | Nginx 同域或子域名部署；飞书微应用配置该地址即可 |

## 5. 详细开发计划

### Phase 0：准备与基础设施（1 周）
1. **需求收敛**：确认移动端 MVP 范围（建议先只做"候选人跟进 + 面试反馈 + 数据看板"）。
2. **UI/UX 设计**：输出移动端高保真原型（Figma/墨刀），重点设计列表页、详情页、审批流程。
3. **工程搭建**：
   - 新建 `mobile/` 目录，配置 Vite + Vue 3 + Vant 4 + TS；
   - 配置路径别名、ESLint/Prettier（与现有项目规范一致）；
   - 配置 `postcss-px-to-viewport` 做屏幕适配；
   - 搭建路由骨架与底部 TabBar（首页、候选人、消息、我的）。
4. **API 复用梳理**：从 `client/src/api/` 中迁移/抽象出移动端需要的接口模块。

### Phase 1：核心功能开发（3 周）
按优先级迭代，每周一个小里程碑：

**Week 1：认证 + 首页 + 个人中心**
- 登录页（复用 JWT 逻辑）
- 首页 Dashboard（今日待办、快捷入口）
- 个人中心（用户信息、退出登录）
- 底部 TabBar 导航

**Week 2：候选人模块**
- 候选人列表（Vant List + Search + Filter）
- 候选人详情（基本信息、工作经历、关联职位）
- 阶段推进（StageRecord 状态流转、备注填写）
- 简历上传（调用 `<input type="file">`，对接现有 `/api/upload`）

**Week 3：面试 + Offer + 职位浏览**
- 面试日程列表
- 面试反馈表单（精简字段，适配手机键盘）
- Offer 列表与审批操作
- 职位列表与详情（只读，支持分享）

### Phase 2：数据看板 + 体验优化（2 周）
1. 数据看板页面（核心 KPI 卡片 + 简化图表）。
2. 下拉刷新、上拉加载、空状态、骨架屏统一处理。
3. 加载状态、网络错误、空数据等边界体验优化。
4. 软键盘弹出时的表单布局适配。
5. 图片预览、文件下载等移动端特有交互。

### Phase 3：联调、测试与部署（2 周）
1. **接口联调**：与后端确认字段裁剪、分页参数、上传兼容性。
2. **测试**：
   - 真机测试（iOS Safari / Android Chrome / **飞书内置浏览器**）；
   - 增加 `mobile/` 的 Vitest 单元测试（重点覆盖 composables 和 stores）；
   - E2E 测试可选（Playwright 支持移动端视口模拟）；
   - **飞书环境专项测试**：免登流程、JSAPI 调用、底部安全区、分享卡片。
3. **部署**：
   - Nginx 增加 `mobile/` 路由（如 `/m/` 子路径或 `m.localhost` 子域名）；
   - 配置 CORS 与 HTTPS（移动端对 HTTP 常有安全限制）。
4. **文档与交付**：更新 README、操作手册、运维文档。

## 6. 工作量预估

**假设团队配置：1 名资深前端 + 0.5 名后端（接口支持）**

| 阶段 | 人天 | 说明 |
|------|------|------|
| Phase 0：准备与基建 | 5 | 含设计稿输出与工程脚手架 |
| Phase 1：核心功能 | 15 | 4 个主模块 × 3～4 天/模块 |
| Phase 2：看板 + 优化 | 10 | 图表适配与体验细节较耗时 |
| Phase 3：联调测试部署 | 10 | 真机兼容性测试占大头 |
| **飞书接入追加** | 6 | 免登接口 + JSAPI 集成 + 飞书环境联调 |
| **缓冲（20%）** | 9 | 需求变更、评审修改、飞书开放平台审批 |
| **总计** | **约 55 人天 / 7～9 周** | 按 1 名前端全职计算 |

**若选择 UniApp（方案 C）**：
- 前端工作量增加约 **30%**（适应 UniApp 生态、多端调试、条件编译）。
- 总周期预估 **9～11 周**。

## 7. 风险点分析与应对

### 7.1 技术风险
| 风险 | 影响 | 应对策略 |
|------|------|----------|
| **Element Plus 与 Vant 样式冲突** | 中 | `mobile/` 作为独立项目构建，不共享 client 的 CSS，彻底隔离 |
| **现有接口返回数据量过大** | 中 | 列表接口增加 `fields` 参数做字段裁剪；详情接口按需分块加载 |
| **富文本在移动端体验差** | 高 | 移动端避免使用富文本编辑器；面试反馈等改用结构化表单 + 文本域 |
| **文件上传在 iOS Safari 限制多** | 中 | 使用 Vant Uploader 组件做兼容性封装；测试覆盖相机/相册/文件三种来源 |
| **飞书 JSAPI 版本兼容** | 中 | 飞书 PC 端与移动端 JSAPI 存在差异，需做环境判断与降级处理 |
| **飞书开放平台审批周期** | 中 | 企业自建应用一般审批较快，但敏感权限（如通讯录）可能需要 1～3 个工作日 |
| **图表在移动端性能问题** | 中 | 数据看板只展示聚合后的关键指标，避免渲染大量原始数据点 |

### 7.2 业务风险
| 风险 | 影响 | 应对策略 |
|------|------|----------|
| **移动端与 PC 端功能边界模糊** | 高 | 严格定义 MVP：移动端主打"查看 + 审批"，复杂录入（如批量导入、字典配置）仍引导至 PC 端 |
| **用户不接受移动端操作习惯** | 中 | 早期邀请 3～5 名核心 HR 做可用性测试，根据反馈快速调整 |
| **权限控制遗漏** | 中 | 复用现有 JWT + role 中间件，前端路由同样配置 `meta.requireAdmin` |

### 7.3 人员/进度风险
| 风险 | 影响 | 应对策略 |
|------|------|----------|
| **前端一人兼顾 PC 需求与 Mobile 开发** | 高 | Phase 1 期间建议暂停 PC 端非紧急需求，或增加 1 名前端支持 |
| **设计稿评审反复** | 中 | 在 Phase 0 预留 2 天评审缓冲；采用设计系统（Vant 组件规范）减少定制化设计 |

---

## 8. 决策建议（下一步行动）

在正式开工前，建议团队就以下 3 个问题达成共识：

1. **是否必须覆盖微信小程序？**
   - 若 **否** → 直接采用 **方案 B（Vue 3 + Vant 独立 H5）**。
   - 若 **是** → 评估 **方案 C（UniApp）**，但需接受约 2 周的额外技术预研。

2. **移动端是否以飞书为主要入口？**
   - 若 **是** → 方案 B 完全可行，需追加 **6 人天** 做飞书免登与 JSAPI 适配；无需开发飞书小程序。
   - 若 **否** → 仍可先输出独立 H5，后续再评估其他平台。

3. **移动端 MVP 范围是否可收敛？**
   - 强烈建议首期只做：**候选人跟进、面试反馈、数据看板**。
   - 职位创建、成员管理、字典配置等复杂功能保留在 PC 端，可显著降低开发周期与风险。

4. **是否需要独立的移动端后端接口（BFF）？**
   - 建议 **第一期不复用现有 API**，通过接口参数优化即可。
   - 若联调中发现大量接口不适配，可在 Phase 2 末评估是否引入轻量 BFF。
